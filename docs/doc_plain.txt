1. Product Vision &amp; Problem Statement
1.1 The Problem
A junior business analyst is assigned a project or task. They may have strong domain knowledge — they understand the process, the people, and the business context — but they do not have a formal BA framework in their head. They do not know which artifacts to produce, which questions to ask, what 'done' looks like, or how to sequence their work. Without guidance, they either miss critical steps, produce incomplete deliverables, or spend significant time researching methodology instead of doing the work.

This problem is not unique to career changers. It affects any analyst who is new to a BA role, working in an unfamiliar domain, or tackling a type of engagement they have not handled before.

1.2 The Solution
F2 is a BA workspace application where AI is embedded throughout the workflow. It acts as a knowledgeable senior BA sitting beside the user. The user brings domain knowledge and business context. F2 provides the methodology, the structure, the artifact templates, and the guided assistance to produce professional-quality BA deliverables.

F2 is not a chatbot wrapper around an LLM. It is a methodology engine with AI layered on top. The BA processes, phases, and artifacts are pre-built and BABOK-aligned. The AI's job is to help the user navigate that structure intelligently — not to invent it on the fly.

1.3 The Core Promise

F2 gives anyone with functional understanding of a business process and basic problem-solving ability the capacity to operate as a full-fledged BA at a high level.
The tool provides the BA brain. The user provides the business knowledge. Together they produce work that is structured, defensible, and BABOK-aligned.

1.4 Portfolio Context
F2 is Flagship 2 in a 6-project BA portfolio built to demonstrate an HR-to-BA career transition. As a flagship, F2 must demonstrate depth — not just that the builder can use an API, but that they understand BA methodology well enough to encode it as product logic.

2. User Model
2.1 Primary User — The Junior BA
The primary user is someone with functional domain expertise who is new to formal BA practice. They have been assigned a project or task and need structured guidance to navigate it professionally.

Attribute
Description
Domain knowledge
Strong — they understand the business context, the process, and the people involved
BA methodology knowledge
Limited to moderate — may know some artifacts but lacks a complete framework
Technical comfort
Moderate — comfortable with business software, forms, and structured tools
Primary need
Guidance on what to do, in what order, and how to produce each deliverable
Fear
Missing something important, producing incomplete work, or looking unprepared
Success metric
Delivers a professional, complete BA artifact package with confidence

2.2 V1 Scope — Solo User Only
Version 1 of F2 is a single-user application. Each user logs in, creates their own projects, and works independently. Collaboration features — shared projects, role-based access, stakeholder review workflows — are explicitly out of scope for V1 and will be addressed in V2.

V1 Constraint
One user. One project at a time. Full persistence across sessions. No multi-user collaboration in this version.

3.2 Scale Tiers
Every project is classified into one of three scale tiers. The tier determines the depth of the roadmap generated. The AI recommends a tier at intake based on what the user describes; the user can override it.

Tier
Description
Initiative
Multi-phase engagement. Multiple stakeholders. Full artifact suite. Timeline: weeks to months. Example: enterprise system implementation, org-wide transformation.
Engagement
Mid-size engagement. A few stakeholders. Targeted artifact set. Timeline: days to weeks. Example: departmental process improvement, new analytics capability.
Request
Small, bounded ask. One or two stakeholders. Minimal artifacts. Timeline: hours to days. Example: document requirements for a new report, fix a specific process step.

4. Intake &amp; Classification System
4.1 Why Intake is the Most Critical Feature
Intake is where the tool earns the user's trust. If the classification is wrong — the wrong engagement type, the wrong scale, the wrong process mapped — then every subsequent step is built on a bad foundation. The intake system must be thorough enough to get this right consistently, while remaining fast enough that the user does not feel interrogated.

4.2 The Two Classification Dimensions
The roadmap template selected for a project is determined by two primary dimensions: Engagement Type and Scale Tier. These are the must-know dimensions — the AI cannot generate a roadmap without them.

4.3 The Ten Intake Dimensions
Beyond the two primary dimensions, eight additional dimensions refine the roadmap and guide the AI's assistance throughout the project. These are captured progressively — not all at once at intake.

Dimension
Description &amp; Why It Matters
1. Engagement Type
What kind of work is this fundamentally? Determines which template is selected. Must-know at intake.
2. Scale / Complexity Tier
How big and complex is this engagement? Determines roadmap depth. Must-know at intake.
3. Trigger / Origin
What caused this work to exist? Problem escalation, opportunity, mandate, directive, or request. Affects stakeholder dynamics and urgency framing.
4. Solution Clarity
How much does anyone already know about the solution? Open exploration vs. formalizing a decided answer changes the BA's job significantly.
5. Stakeholder Landscape
How complex is the human side? Single stakeholder vs. conflicting multi-party engagements require very different approaches.
6. Domain / Functional Area
What part of the business does this touch? Affects language, examples, and artifact relevance.
7. Deliverable Expectation
Has anyone specified what needs to be produced? Open vs. pre-defined deliverables changes how the roadmap is scoped.
8. Timeline / Urgency
How much time does the BA have? A scope filter — the tool should flag when timeline and roadmap depth are misaligned.
9. Data / Evidence Availability
Is there existing material to work from? Determines whether gap analysis becomes critical early and whether uploads should be prompted.
10. Change Impact
How disruptive is the expected outcome? Determines whether change management artifacts are included in the roadmap.

4.4 Tiered Capture Logic
Not all dimensions are captured at the same time. The intake system uses a three-tier capture approach to avoid overwhelming the user on Day 1.

Tier
Dimensions
When Captured
Must-Know
Engagement Type, Scale Tier, Trigger / Origin, Solution Clarity
At intake — before roadmap is generated
Should-Know Early
Stakeholder Landscape, Deliverable Expectation, Timeline / Urgency
First session — within first module or two
Emerges During Work
Domain / Functional Area, Data / Evidence Availability, Change Impact
As the engagement develops — AI prompts when relevant

4.5 Intake Experience
The intake experience is conversational on the surface but structured underneath. The AI asks open-ended questions and listens to natural language input — the user can paste an email, describe a problem verbally, or paste a brief. The AI is systematically working through the must-know dimensions under the hood before it commits to a classification.

Once the AI has enough signal, it presents its recommended classification — Engagement Type + Scale Tier — with a plain-language explanation of why. The user can accept the recommendation or override it. The roadmap is then generated from the corresponding pre-built template.

Living Roadmap Principle
The roadmap is not fully locked at intake. It is a living document. As new dimensions become clear during the work, the AI can suggest module additions, flag scope mismatches, and warn on dependency violations. This matches how real BA engagements actually evolve.

5. UX Model — Navigator + Co-Pilot Hybrid
5.1 The Hybrid Model
F2 uses a hybrid UX model that combines two distinct modes of interaction depending on where the user is in their work.

Mode
Where It Applies
What It Does
Navigator
Project level — the roadmap view
The user always sees the full roadmap. All modules are visible. Progress is tracked. The user understands at a glance where they are, what is ahead, and what is blocked. Free-form navigation — any module can be entered in any order.
Co-Pilot
Module level — inside a specific module
When the user dives into a module, AI takes the wheel. It asks questions, fills in content based on answers, flags gaps, suggests improvements, and guides the user to completion. The user is doing the work; the AI is the expert looking over their shoulder.

5.2 Navigation Principle
Navigation is free-form. Users can access any module from any project without being forced through a rigid sequence. The AI will warn when a user attempts to work on a module whose dependencies are not yet complete — but it will not block them. The user is always in control.

5.3 The Roadmap as the Home Base
When a user opens a project, they land on the roadmap view. This is their home base for the entire engagement. The roadmap shows every module in the project's sequence, the status of each, and the overall completion progress. It is the map that is always visible — even when the GPS (co-pilot mode) is talking.

6. Product Architecture
6.1 Three-Layer Architecture
F2 is built on three distinct layers. Each layer has a clear responsibility. Together they form a modular workflow engine.

Layer
Description
Layer 1 — Module Library
A fixed set of 52 pre-built BA modules, each representing a discrete unit of work. BABOK-aligned. Defined by the builder. Static in V1. Each module has a name, knowledge area, description, inputs required, and artifacts produced.
Layer 2 — Roadmap Templates
24 pre-built combinations of modules assembled for each Engagement Type + Scale Tier combination. These are the best-practice sequences for each scenario. Static in V1. The template is the starting configuration for a new project.
Layer 3 — Live Roadmap
The instance created for a specific project. Starts as a copy of the selected template. Fully editable — modules can be added, removed, or reordered. Belongs to the project. Can diverge from the template as the engagement evolves.

6.2 AI Intelligence Layer
The AI operates on top of the three-layer architecture. It does not generate the methodology — it navigates it intelligently. The AI's responsibilities are:

Intake classification — read natural language input and map it to an Engagement Type + Scale Tier
Dimension extraction — systematically work through intake dimensions during the intake conversation
Roadmap suggestions — recommend module additions when new dimensions reveal gaps
Dependency warnings — flag when a user attempts a module whose prerequisites are incomplete
Scope conflict alerts — flag when timeline and roadmap depth are misaligned
Co-pilot assistance — guide the user through each module with targeted questions and content generation
Gap analysis — review completed artifacts and flag what is weak, missing, or inconsistent

6.3 Data Model Sketch
The following tables form the core data model for F2. This is a design sketch — exact schema will be finalized during the technical build phase.

Table
Purpose &amp; Key Fields
users
Auth table. user_id, email, created_at. Managed by Supabase Auth.
projects
One row per project. project_id, user_id, name, description, engagement_type, scale_tier, status, created_at, updated_at.
project_dimensions
Intake dimension values for each project. project_id, dimension_name, dimension_value, captured_at. One row per dimension per project.
modules
The module library. module_id, name, knowledge_area, description, typical_inputs, typical_outputs. Static reference table.
roadmap_templates
Pre-built template definitions. template_id, engagement_type, scale_tier, module_sequence (ordered array of module_ids).
project_roadmap
The live roadmap for a project. roadmap_item_id, project_id, module_id, sequence_order, status (not_started / in_progress / complete / skipped), notes, created_at, updated_at.
artifacts
Outputs produced within modules. artifact_id, project_id, module_id, artifact_type, content (text / structured JSON), version, created_at, updated_at.

Module Name
Description

Document in detail what the system, process, or solution must do — specific behaviors, functions, and capabilities required.

Process Modeling — Future State

Use Case Development

Data Requirements Definition
Define what data the solution requires — sources, formats, transformations, storage, and governance requirements.
Business Rules Documentation
Document the logic, conditions, and constraints that govern how the solution must behave under various circumstances.
Acceptance Criteria Definition
Define the specific, testable conditions that must be met for each requirement to be considered successfully delivered.
Solution Prototyping &amp; Wireframing
Produce low-fidelity representations of the solution for stakeholder validation before detailed design or development begins.
Requirements Verification
Internal quality check confirming that requirements are complete, consistent, unambiguous, and traceable before stakeholder review.
Requirements Validation
Stakeholder-facing confirmation that documented requirements accurately reflect actual business needs and intended outcomes.

Document organizational or environmental constraints that are limiting the solution's ability to deliver full expected value.

openai Python package

Reason Rejected

Streamlit is the established stack for this portfolio
SQLite
Persistence must survive deployment — Neon DB required

Users log in, projects must persist across sessions

OpenAI first — Anthropic considered for V2
React
Outside the Python/Streamlit stack

10. Prioritized Build Sequence
V1 Build Phases
The following sequence is recommended for V1 development. Each phase must be complete and stable before the next begins. Prompt quality is treated as a first-class deliverable — AI prompts should be iterated and tested in isolation before UI is built around them.

Phase
Deliverable
Priority
Phase 1
Project scaffold — auth, project creation, database schema, navigation shell
Foundation
Phase 2
Module library and template data — all 52 modules and 24 templates seeded into the database
Core data
Phase 3
Intake &amp; classification — conversational intake UI, AI classification logic, dimension capture, roadmap generation from template
Critical path
Phase 4
Roadmap navigator view — project home, module list, status tracking, progress visualization
Core UX
Phase 5
Co-pilot module experience — first 3-4 modules built with full AI co-pilot interaction (Problem Statement, Stakeholder Interviews, Current State Assessment, Requirements Elicitation)
Core value
Phase 6
Artifact generation — structured output for key artifacts (FRD, user stories, acceptance criteria, stakeholder register)
Deliverable quality
Phase 7
Gap analysis — AI review of completed artifacts, flagging weaknesses and inconsistencies
Intelligence layer
Phase 8
Traceability matrix — automated linkage of requirements to user stories and test cases
Intelligence layer
Phase 9
Export &amp; packaging — deliverable export in professional format
Completion
Phase 10
Polish, testing, deployment
Launch readiness

V1 Scope Boundaries
The following are explicitly out of scope for V1:

Multi-user collaboration — shared projects, role-based access, stakeholder review workflows
Module library editing — adding, modifying, or removing modules via UI
Custom template creation — users cannot create their own roadmap templates
Anthropic API — OpenAI GPT-4o only in V1
Mobile optimization — desktop/web first
Integrations — no Jira, Confluence, or other external tool integrations in V1

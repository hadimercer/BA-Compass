1. Purpose & Framing

This stakeholder register treats the build and deployment of BA Compass (F2) as a formal BA engagement. That framing is intentional: the register itself is a portfolio demonstration of BA practice applied to a real-world scenario — in this case, Hadi's own career transition project.

The register identifies all parties with a stake in the successful delivery of BA Compass, documents their interest, influence, expectations, and risks, and specifies an appropriate engagement approach for each. It follows the BABOK v3 stakeholder analysis framework.

Engagement framing: The "client" is Hadi Mercer as the product owner and career-transition candidate. The "deliverable" is a fully functional, portfolio-quality BA Co-Pilot application. The "business outcome" is a demonstrated BA skill set that accelerates a career transition from HR/data to Business Analysis.

3. Stakeholder Profiles

Each profile documents the stakeholder's role, interests, influence, expectations, potential risks they pose to the engagement, and the engagement approach applied.

🧑‍💻

SH-01 — Hadi Mercer

Builder · Product Owner · Primary End User · Portfolio Candidate
Interest: High Power: High Manage Closely
Primary Interest
Deliver a portfolio-quality, production-grade application that demonstrably validates a career transition from HR/data to Business Analysis. Must be deployable, defensible, and impressive to technical evaluators.
Influence
Full authority over product scope, design decisions, technology choices, and build sequence. Has final approval over all pre-build documents. Can halt, pivot, or change direction at any point.
Expectations
High visual quality bar; professional-grade documentation; code that works first time; explanations before execution; no wholesale file regenerations; PowerShell syntax for all terminal commands.
Constraints
Working on Windows. Learning Python and SQL — implementation is AI-assisted. Time-constrained: portfolio must be job-application-ready. Budget-conscious: Streamlit Community Cloud (free), Neon DB (free tier).
Risks Posed
Scope creep driven by domain expertise — deep knowledge may generate endless "what if we also..." additions. Risk of perfectionism stalling delivery. Learning curve on Python may slow debugging.
Engagement Approach
Explain before executing. Seek approval at each document milestone before proceeding. Flag scope risks explicitly when they arise. Provide targeted fixes, not rewrites. Treat every output as a portfolio artefact.
Success Criteria (from Hadi's perspective)
BA Compass is live on Streamlit Cloud, passes a portfolio walkthrough without visible flaws, and clearly demonstrates BABOK methodology embedded in product logic — not just an LLM chat wrapper. Hiring manager or BA community reviewer responds with a credibility-level reaction, not a polite "interesting project."
👔

SH-02 — Hiring Managers

BA / Product / Data roles · Financial services, consulting, technology sectors · Canada-primary
Interest: High Power: High Manage Closely
Primary Interest
Assess whether Hadi's BA skills are credible enough to justify hiring relative to candidates with direct BA job titles. They are evaluating depth of methodology knowledge, not just tool familiarity.
Influence
High — they are the ultimate decision-makers in the engagement outcome (employment). Their evaluation criteria implicitly define the quality bar for the portfolio.
Expectations
Evidence of structured thinking. BABOK alignment. A product that solves a real problem. Professional documentation that holds up to scrutiny. Code that actually runs — not just screenshots. Clear articulation of design decisions.
Key Concerns
Is this genuinely the candidate's work, or is it all AI-generated? Does the candidate understand what they built and why? Can they walk through methodology decisions without reading from a script?
Risks Posed
May dismiss portfolio if it appears overly AI-dependent. May not review documentation if application demo is weak. Short attention window — first 90 seconds of a walkthrough are decisive.
Engagement Approach
Design every pre-build document so Hadi can speak to it fluently in an interview. Ensure the application's UI surface communicates methodology without explanation. Produce a 2-minute demo script as a final artefact.
🔍

SH-03 — Portfolio Reviewers

BA community (LinkedIn, IIBA chapters) · Academic / bootcamp assessors · Peer reviewers
Interest: High Power: Medium-High Manage Closely
Primary Interest
Evaluate the rigour and authenticity of the BA methodology demonstrated. Interested in whether the BABOK knowledge is applied correctly, not just referenced nominally.
Influence
Medium-high — LinkedIn endorsements, referrals, and community credibility materially affect job search outcomes. A strong IIBA community reaction multiplies the portfolio's reach.
Expectations
Correct BABOK terminology. Traceability between requirements and deliverables. Demonstrated understanding of elicitation, analysis, and solution evaluation — not just requirements documentation. Pre-build artefacts that a senior BA would actually produce.
Risks Posed
May identify BABOK misapplication if terminology is used loosely. Risk of "this is a tech project, not a BA project" dismissal if the methodology layer is not visible enough.
Engagement Approach
Ensure all 52 modules are accurately categorised by BABOK knowledge area. Use BABOK v3 vocabulary throughout documentation. Make the methodology engine the headline, not the AI integration.
📋

SH-04 — Recruiters

Technical and generalist recruiters · Agency and in-house · BA, Product, Data roles
Interest: Medium Power: Medium-Low Keep Informed
Primary Interest
Quickly assess whether Hadi is placeable for BA-adjacent roles. Recruiters are keyword-matchers at first pass — they will skim the portfolio for recognisable signals: BABOK, SQL, Python, Streamlit, data analysis.
Influence
Medium-low — gatekeepers to opportunities but not final decision-makers. A recruiter who "doesn't get it" is a miss, not a failure. The portfolio is primarily built for hiring managers, not recruiters.
Expectations
A live link to the application. A clear one-paragraph description of what the tool does and who it's for. Recognisable keywords in the project description. Evidence of working software.
Engagement Approach
Ensure the README and portfolio landing page has a recruiter-scannable summary: tool name, problem solved, tech stack, and a live link. The documentation depth is for managers; the headline is for recruiters.
🎓

SH-05 — Junior BAs (Target User Persona)

0–3 years BA experience · Recent grads / career changers · Working without senior mentors
Interest: High Power: Low Keep Informed
Primary Interest
Get a tool that actually helps them navigate a new BA engagement without a senior mentor standing over their shoulder. They want structure, guidance, and outputs they can be proud of.
Influence
Low in V1 — no formal user research or feedback loop is built into the product roadmap yet. They represent the design target, not active participants in the build.
Expectations
A tool that explains why, not just what. Methodology embedded in prompts and guidance, not assumed. An interface that doesn't look like a student project. Outputs they can submit to a client without embarrassment.
Pain Points Addressed
Blank-page anxiety on new engagements. Uncertainty about which BABOK areas apply. Inconsistent artifact quality. No way to know if they've done enough. No mentor to sanity-check their work.
Engagement Approach
Keep this persona central during all UX and prompt design decisions. Hadi's own experience as a methodical, analytical thinker formalising a BA skill set makes him the ideal empathy source for this persona.
🤖

SH-06 — OpenAI (API Provider)

External service provider · GPT-4o · Critical dependency
Interest: Low Power: High Keep Satisfied
Nature of Relationship
OpenAI has no interest in BA Compass specifically but holds high power via API reliability, pricing policy, and model availability. A GPT-4o deprecation or pricing change would require significant product rework.
Key Risks
API unavailability during a portfolio demo. Unexpected rate limit behaviour under concurrent use. Model output quality degradation across versions. Prompt format changes in future SDK versions.
Engagement Approach
Use the openai Python package pinned to a tested version. Implement graceful error handling on all API calls. Store all prompts as editable config, not hardcoded strings. Monitor OpenAI changelog as part of maintenance.
Contractual / Terms
Usage must comply with OpenAI Terms of Service. API key must never be committed to version control. Portfolio demo usage must stay within free-tier or paid-tier limits as applicable.
🏗️

SH-07 — Infrastructure Providers

Neon DB · Streamlit Community Cloud · GitHub
Interest: Low Power: Medium Monitor
Neon DB
Provides PostgreSQL hosting on the free tier. Risk: connection limits on free tier may affect concurrent demo usage. Engagement: monitor connection pooling; use a lightweight ORM or direct psycopg2 with proper connection management.
Streamlit Community Cloud
Provides free deployment. Risk: cold-start latency on infrequently accessed apps may create a poor first impression during a demo. Engagement: keep app lean; use st.cache_data and st.cache_resource appropriately.
GitHub
Version control and Streamlit Cloud deployment trigger. Risk: accidental commit of secrets if .gitignore is not set up correctly from Day 1. Engagement: configure .gitignore in project scaffold before any other files are committed.
Engagement Approach
All three are "monitor" tier — no active engagement required beyond setup. Establish monitoring checks before launch: verify Neon connection health, confirm Streamlit app loads within 5s, confirm GitHub secrets protection is active.

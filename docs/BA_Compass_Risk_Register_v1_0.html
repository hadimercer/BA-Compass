1. Purpose & Scope

This risk register documents all identified risks associated with the design, build, and deployment of BA Compass (F2). It covers three risk categories: product risks (relating to the quality and correctness of the application's outputs), technical risks (relating to implementation, infrastructure, and API dependencies), and portfolio risks (relating to the strategic objectives of the project as a career-transition portfolio piece).

Each risk is rated by likelihood and impact, assigned a composite risk rating, and given a mitigation strategy and contingency response. The register is produced as the final pre-build document — informed by all prior analysis — and is intended to remain active as a reference throughout the build phase.

Risk ratings use a 3×3 likelihood/impact matrix (Low / Medium / High). Composite ratings: High × High = Critical; High × Medium or Medium × High = High; Medium × Medium = Medium; all others = Low. Owner is Hadi Mercer for all risks in V1 (solo engagement).

3. Risk Detail — Product Risks

Product risks relate to the quality, accuracy, and user trust of BA Compass's core outputs — particularly the classification engine and the co-pilot artifact generation.

R-01 Classification accuracy insufficient for user trust
Critical Likelihood: High Impact: High
Description
The intake classification engine is the most critical feature in BA Compass. If GPT-4o's engagement type and scale tier recommendation is frequently wrong, implausible, or poorly reasoned, users will lose confidence in the entire roadmap generated from it. Classification error at the foundation cascades into every downstream output.
Trigger Conditions
Ambiguous intake responses; edge-case engagement types; insufficient prompt specificity; model hallucination on classification rationale
Mitigation
Treat classification prompt as a first-class engineering deliverable. Develop and iterate the prompt in isolation before building UI around it. Test against all 8 engagement types × 3 scale tiers with synthetic inputs. Make AI reasoning transparent in the UI — users who can see why a classification was made are more likely to trust it or make an informed override.
Contingency
User override is a V1 requirement — not a nice-to-have. If classification accuracy cannot be reliably improved, the product remains usable because the user is always in control. Reclassification must also be available at any point post-intake.
R-02 Co-pilot output quality inconsistent across modules
High Likelihood: High Impact: Medium
Description
With 52 modules, each requiring its own prompt, the co-pilot experience will almost certainly be uneven across the module library at V1 launch. Some modules (e.g. stakeholder register, process flow documentation) will produce strong outputs; others (e.g. benefits realisation, regulatory impact analysis) may produce generic or shallow drafts.
Trigger Conditions
Prompts written hastily for less-familiar modules; modules with highly context-dependent outputs; modules where BA methodology is nuanced and harder to encode
Mitigation
Prioritise prompt quality for the 15–20 modules most likely to be used in a portfolio demo. Build a prompt review phase into the build sequence — each prompt tested with 3+ synthetic inputs before being considered stable. Use a consistent prompt template structure across all modules to reduce variance.
Contingency
Weaker modules can be labelled "beta" in V1 UI without impacting the overall product impression. A strong core module set (10–15 modules) is sufficient for a compelling demo. Flag weak modules for iteration post-launch.
R-03 Scope creep expanding V1 beyond buildable limits
Critical Likelihood: High Impact: High
Description
Hadi's deep domain knowledge is an asset during product definition and a risk during build. The same expertise that produces a rigorous 52-module library can generate a continuous stream of "what if we also added..." additions that push V1 scope beyond what can be built, tested, and polished within a portfolio timeline.
Trigger Conditions
Discovery of edge cases during build; comparison with existing BA tools; feedback during document review; iterating too deeply on a single phase before completing others
Mitigation
The Product Definition Document is locked — it is the authoritative scope boundary. Any new idea encountered during build is logged in a "post-V1 ideas" backlog, not added to V1. Build phase follows the 10-phase sequence strictly — no phase begins until the prior is stable. Phase gate reviews enforce discipline.
Contingency
If scope creep has already entered a phase, conduct an explicit descope exercise: list every in-progress addition, evaluate each against the V1 definition, remove anything not in the original spec. A working V1 with clean scope is a stronger portfolio piece than a partially-built V1.5.
R-04 BABOK alignment superficial or incorrect
High Likelihood: Medium Impact: High
Description
BA Compass's core claim is that it encodes BABOK methodology, not just wraps an LLM. If the module library uses BABOK terminology superficially, maps modules to the wrong knowledge areas, or the co-pilot prompts do not reflect genuine BA practice, a knowledgeable reviewer (IIBA member, senior BA hiring manager) will identify this immediately — and it will invalidate the entire portfolio narrative.
Trigger Conditions
Module knowledge area assignments made by convention rather than BABOK v3 reference; prompt language that sounds like BA methodology but misses key concepts; reviewers with deep IIBA/BABOK knowledge
Mitigation
Map all 52 modules to BABOK v3 knowledge areas using the BABOK Guide directly as reference, not from memory. Have at least one section of the documentation reviewed by an IIBA-certified peer if accessible. Ensure co-pilot prompts reference the correct BABOK tasks and techniques for each module type.
Contingency
If a misalignment is identified during review, correct it in the seed data before launch — the module library is a static data file, not compiled code. A post-launch patch is possible without touching application logic.
R-05 Gap analysis produces false confidence — misses real gaps
High Likelihood: Medium Impact: High
Description
The gap analysis feature is a high-value differentiator. However, if the AI reviews artifacts and fails to flag genuine gaps — either because the prompt is under-specified, the AI is too agreeable, or the engagement context is incomplete — users may submit deliverable packages that appear validated but contain real weaknesses. This is worse than having no gap analysis at all.
Trigger Conditions
Gap analysis prompt too generic; AI lacks sufficient engagement context at review time; artifacts are thin but not obviously incomplete; model optimised for positive framing
Mitigation
Gap analysis prompt must be explicitly instructed to be critical and specific — not to validate, but to challenge. Include all 10 intake dimensions in the gap analysis context. Test the prompt against known-incomplete artifact sets to verify it flags real gaps. Use a structured output format that requires the AI to assess each BABOK knowledge area independently.
Contingency
Add a disclaimer in the UI: gap analysis is an AI-assisted review, not a guarantee of completeness. Frame it as "a second pass before you submit" — valuable, but not a substitute for professional judgment. This is also an honest statement of the product's current capability.

4. Risk Detail — Technical Risks

Technical risks relate to the reliability of the infrastructure, API dependencies, and implementation decisions that underpin the application.

R-06 OpenAI API unavailability during portfolio demo
Medium Likelihood: Low Impact: High
Description
OpenAI API outages are infrequent but not impossible. An outage during a live portfolio walkthrough with a hiring manager or at a critical review moment would make the core features of BA Compass non-functional and leave a deeply negative impression.
Mitigation
Implement graceful error handling on all API calls — clear, professional user-facing messages rather than raw error traces. Monitor OpenAI's status page at status.openai.com before scheduled demos. Consider caching classification and co-pilot responses for a demo project so a recorded walkthrough is always available as backup.
Contingency
Maintain a screen recording of a complete demo walkthrough as a fallback. If live demo is not possible, a polished recording with commentary is a credible alternative for async portfolio review.
R-07 Neon DB free-tier connection limits cause failures
Medium Likelihood: Medium Impact: Medium
Description
Neon DB's free tier has connection and compute limits. Streamlit creates a new Python process per user session, which can exhaust connection pools under concurrent load or if connections are not properly closed. For a solo-user V1 application this is unlikely, but poor connection management could cause intermittent failures.
Mitigation
Use st.cache_resource to manage a single shared database connection per Streamlit instance. Implement connection pooling via psycopg2 connection pool or use Neon's built-in connection pooler (PgBouncer). Always close connections in finally blocks. Avoid opening connections inside loops.
Contingency
If free-tier limits become a problem post-launch, Neon's paid tier is low-cost and can be upgraded without schema or connection string changes. Budget impact is minimal.
R-08 Streamlit cold-start latency creates poor demo impression
High Likelihood: High Impact: Medium
Description
Streamlit Community Cloud apps that have not been accessed recently go to sleep and require a cold start on next visit — typically 10–30 seconds of a loading spinner before the app responds. For a hiring manager clicking a portfolio link, this creates an immediate negative first impression and may result in them closing the tab before the app loads.
Mitigation
Add a "wake up" ping to the app before any scheduled demo. Keep the app lightweight to minimise cold-start time. Use st.cache_data and st.cache_resource throughout to minimise re-computation on subsequent interactions. Ensure the loading state displays a purposeful, branded screen rather than a blank white page.
Contingency
Pre-warm the app by visiting it ~2 minutes before any demo. For portfolio links shared on LinkedIn or in applications, include a note that the app may take up to 30 seconds to load on first visit — sets expectations and prevents abandonment.
R-09 API key or secrets accidentally committed to GitHub
High Likelihood: Medium Impact: High
Description
A committed API key or database credential on a public GitHub repository creates an immediate security incident — the key must be rotated, and the commit history may expose the secret even after removal. For a portfolio repository reviewed by hiring managers, this would be professionally embarrassing and potentially costly if the key is misused before rotation.
Mitigation
Configure .gitignore before the first commit — .env, secrets.toml, and any local config files must be listed. All secrets stored in Streamlit Secrets only — never in code or config files that could be committed. Enable GitHub secret scanning on the repository. Use a pre-commit check to scan for key patterns before pushing.
Contingency
If a secret is committed: immediately rotate the key in the relevant provider dashboard, use git filter-branch or BFG Repo Cleaner to purge the secret from history, and force-push the clean history. Do not rely on a simple deletion commit — the secret remains in history.
R-10 OpenAI pricing or model deprecation forces rework
Low Likelihood: Low Impact: Medium
Description
GPT-4o may be deprecated or significantly repriced. For a portfolio application with modest usage, pricing changes are unlikely to be material. Model deprecation would require updating the model string and potentially re-tuning prompts for a successor model.
Mitigation
Store the model name as a config constant, not a hardcoded string. All prompts stored as editable config — swapping models requires changing one constant and re-testing prompts, not rewriting application logic.
Contingency
Monitor OpenAI's model deprecation schedule. GPT-4o deprecation typically provides 6+ months notice. A prompt re-test against a successor model is a few hours of work if prompts are well-structured.

5. Risk Detail — Portfolio Risks

Portfolio risks relate to the strategic objective of BA Compass as a career-transition portfolio piece. These risks are distinct from technical failure — the application can work perfectly and still fail to achieve its portfolio purpose if the following risks are not managed.

R-11 Portfolio dismissed as AI-generated, not Hadi's own work
High Likelihood: Medium Impact: High
Description
An evaluator who sees a polished, comprehensive BA application may conclude that it was produced entirely by AI with minimal intellectual contribution from the candidate — particularly in a hiring climate where AI-generated portfolios are common. If Hadi cannot fluently explain every design decision, the dismissal is validated.
Mitigation
Every pre-build document must be something Hadi can speak to fluently in an interview — the process of producing them is as valuable as the documents themselves. Prepare a 2-minute walkthrough narrative that articulates methodology decisions, not just features. The portfolio narrative emphasises that AI handles implementation; Hadi provides the BA methodology expertise that the AI cannot encode without him.
Contingency
If challenged, be direct: "I built this with AI as a coding tool, the same way a developer uses Stack Overflow or a compiler. The product logic — the 52-module BABOK library, the 8 engagement types, the intake dimension model — came from my BA knowledge. The AI couldn't have done that without me specifying it." That is a credible and honest answer.
R-12 Build timeline extends beyond hiring window
High Likelihood: Medium Impact: High
Description
F2 is a large, complex build. If the build phase runs long — due to scope creep, technical challenges, or competing priorities — the portfolio may not be ready when active hiring conversations are underway. A partially built application is worse than no application in a portfolio context.
Mitigation
The 10-phase build sequence defines a minimum viable completion point: phases 1–5 (scaffold, module data, intake, roadmap navigator, co-pilot) deliver the core value proposition and are demo-able as a complete product. Phases 6–10 add depth. Target phases 1–5 as the go/no-go milestone for active job applications. Do not wait for phases 6–10 before putting the portfolio in front of hiring managers.
Contingency
If timeline pressure is acute, S1–S4 (already live) are sufficient to demonstrate technical capability. F2 phases 1–5 are sufficient to demonstrate BA methodology depth. Frame an in-progress F2 as "currently in active development" with a clear build log — this demonstrates momentum, not incompleteness.
R-13 Product complexity outweighs demonstrated value in review
Low Likelihood: Low Impact: Medium
Description
BA Compass is architecturally complex — 52 modules, 24 templates, 10 intake dimensions, multi-phase workflow. A reviewer who cannot quickly understand what the product does and why may be impressed by its ambition but unable to articulate its value — which makes it harder to advocate for Hadi internally.
Mitigation
The application's landing page and the first 90 seconds of any demo must communicate the core value proposition in plain language: "A junior BA assigned a new engagement doesn't know where to start. BA Compass tells them." Everything else is supporting detail. Design the demo flow to hit this message first, then reveal complexity progressively.
Contingency
Prepare a 30-second elevator pitch that reduces BA Compass to one sentence. Use it at the start of every walkthrough, every LinkedIn post, every README. Complexity is a positive signal once the value proposition lands.
R-14 UI quality fails to meet portfolio visual standard
Medium Likelihood: Medium Impact: Medium
Description
Hadi has high visual quality standards — explicitly referencing S4 (Lens) as the visual benchmark. Default Streamlit widgets and layout are unlikely to meet this bar without significant customisation via CSS injection and custom components. Building to this standard takes longer than building to default Streamlit quality.
Mitigation
Invest in the visual system during the scaffold phase — establish custom CSS, component patterns, and the dark-theme colour palette before building any feature screens. A consistent visual system applied consistently is more efficient than retrofitting styles screen by screen. Use S4 (Lens) as a living reference during build.
Contingency
If time pressure forces a choice between UI polish and feature completeness, prioritise the screens that appear in the demo flow. A demo that hits 5 polished screens is stronger than a full application with inconsistent quality throughout.

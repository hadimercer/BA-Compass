1. Overview

This document presents the UML Use Case Diagram for BA Compass — Business Analysis Co-Pilot. The diagram identifies all actors that interact with the system, the use cases each actor participates in, and the system boundary within which those use cases operate.

The use cases are derived directly from the Functional Requirements Document (FRD v1.0). Each use case maps to one or more functional requirements and represents a distinct, observable interaction between an actor and the system.

1.1 Actors

Three actors interact with the BA Compass system.
ActorTypeDescription
User (Junior BA) Primary / Human The business analyst using BA Compass. Initiates all workflows, provides domain knowledge, reviews and edits AI-generated content, and makes all decisions.
AI Engine (GPT-4o) System / External The OpenAI GPT-4o model accessed via API. Performs classification, guides module conversations, generates artifact content, runs gap analysis, and provides recommendations. Never acts autonomously — always in response to user actions or system triggers.
Database (Neon DB) System / Internal PostgreSQL database via Neon DB. Stores all persistent data: user accounts, projects, dimensions, roadmaps, conversation history, and artifacts. Provides data to all other actors.

1.2 System Boundary

The system boundary encompasses all functionality delivered by the BA Compass web application. External actors (AI Engine, Database) sit outside the boundary but participate in use cases within it. The User interacts with the system through the Streamlit web interface.

3.1 Authentication & Access

Register Account
ActorsUser, Database
DescriptionA new user creates an account by providing an email address and password. The system validates the input, hashes the password with bcrypt, and stores the credentials in Neon DB.
PreconditionsUser is on the registration page. No account exists for the email.
PostconditionsA new user record exists in the database. The user is logged in and redirected to the dashboard.
Related FRsFR-101
Log In / Log Out
ActorsUser, Database
DescriptionA returning user authenticates with their email and password. The system verifies credentials against Neon DB and creates a session token. Logout terminates the session.
PreconditionsUser has a registered account.
PostconditionsLogin: authenticated session created. Logout: session terminated, user returned to login page.
Related FRsFR-102, FR-105, FR-106
Maintain Session
ActorsDatabase
DescriptionThe system maintains the user's authenticated session across page reloads and Streamlit reruns. Session tokens are validated on each request.
PreconditionsUser is authenticated.
PostconditionsSession persists until explicit logout or expiry.
Related FRsFR-104

3.2 Project Management

Create Project
ActorsUser, Database
DescriptionThe user creates a new project by providing a name and free-text description. The project is persisted to the database with status and timestamps.
PreconditionsUser is authenticated and on the dashboard.
PostconditionsA new project record exists. The user is directed to the intake flow.
Related FRsFR-201, FR-207
View Project List
ActorsUser, Database
DescriptionThe dashboard displays all projects belonging to the authenticated user with key metadata: name, engagement type, scale tier, status, and last modified date.
PreconditionsUser is authenticated.
PostconditionsProject list is displayed with current data.
Related FRsFR-202, FR-208
Open Project
ActorsUser, Database
DescriptionThe user selects a project from the list and enters the project workspace. The roadmap view loads as the default landing page.
PreconditionsAt least one project exists.
PostconditionsProject workspace is displayed with roadmap view.
Related FRsFR-203, FR-501
Edit Project Details
ActorsUser, Database
DescriptionThe user modifies a project's name or description after creation.
PreconditionsProject exists and is open.
PostconditionsUpdated metadata is persisted. Last modified timestamp updated.
Related FRsFR-204, FR-207
Delete Project
ActorsUser, Database
DescriptionThe user permanently removes a project after explicit confirmation. All associated data (dimensions, roadmap, artifacts) is deleted.
PreconditionsProject exists. User confirms deletion.
PostconditionsProject and all child records are removed from the database.
Related FRsFR-205

3.3 Intake & Classification

Describe Assignment
ActorsUser
DescriptionThe user provides natural language input describing their assignment. This can be typed text, a pasted email, a brief, or any free-text description of the work they have been given.
PreconditionsA new project has been created and the intake flow has started.
PostconditionsThe AI has raw input to begin classification signal extraction.
Related FRsFR-301
Extract Classification Signals
ActorsAI Engine
DescriptionThe AI analyzes the user's natural language input to identify signals for the four Must-Know dimensions: Engagement Type, Scale Tier, Trigger/Origin, and Solution Clarity. This happens internally before any classification is presented.
PreconditionsUser has provided initial input.
PostconditionsThe AI has a confidence assessment for each Must-Know dimension.
Related FRsFR-302
Ask Follow-Up Questions
ActorsAI Engine, User
DescriptionWhen the AI lacks sufficient signal for one or more Must-Know dimensions, it asks targeted follow-up questions. No more than two questions per turn. Questions are non-repetitive and build on what has already been captured.
PreconditionsInitial input has been analyzed. One or more dimensions lack sufficient signal.
PostconditionsAdditional signal captured. Classification confidence improved.
Related FRsFR-303, FR-315
Present Classification Recommendation
ActorsAI Engine
DescriptionOnce the AI has sufficient confidence, it presents its recommended Engagement Type and Scale Tier with a plain-language explanation of why each was selected.
PreconditionsAll Must-Know dimensions have sufficient signal.
PostconditionsThe user sees the recommendation and can act on it.
Related FRsFR-304, FR-305, FR-306
Accept or Override Classification
ActorsUser
DescriptionThe user reviews the AI's recommendation and either accepts it or overrides it by selecting a different engagement type and/or scale tier. This triggers roadmap generation.
PreconditionsClassification recommendation has been presented.
PostconditionsEngagement type and scale tier are confirmed. Roadmap generation is triggered.
Related FRsFR-307, FR-308
Capture Intake Dimensions
ActorsUser, AI Engine, Database
DescriptionThe system captures all ten intake dimensions across three tiers: Must-Know at intake, Should-Know Early in the first session, and Emerges During Work progressively. Each value is persisted with a capture timestamp.
PreconditionsProject exists. Intake has started.
PostconditionsDimension values are stored in project_dimensions with timestamps.
Related FRsFR-309–313

3.4 Roadmap & Navigation

Generate Live Roadmap from Template
ActorsDatabase
DescriptionThe system copies the module sequence from the matching roadmap template (Engagement Type × Scale Tier) into the project's live roadmap. All modules start with status Not Started.
PreconditionsClassification has been confirmed by the user.
PostconditionsA live roadmap exists in project_roadmap with all template modules.
Related FRsFR-401–403
View Roadmap (Navigator Mode)
ActorsUser, Database
DescriptionThe user sees the full roadmap with every module displayed, showing name, knowledge area, status, and overall completion percentage. Modules are visually distinguished by status. This is the project home base.
PreconditionsA live roadmap exists for the project.
PostconditionsRoadmap is rendered with current status data.
Related FRsFR-501–504, FR-508
Add / Remove / Reorder Modules
ActorsUser, Database
DescriptionThe user modifies their live roadmap by adding modules from the library, removing existing modules, or changing the sequence order. Removed modules' artifacts are preserved as orphaned.
PreconditionsA live roadmap exists.
PostconditionsLive roadmap is updated. Changes persisted to database.
Related FRsFR-404–406
Track Module Status
ActorsUser, Database
DescriptionModule statuses update as the user works: auto-transition to In Progress on entry, manual marking as Complete or Skipped. Re-entering a completed module reverts it to In Progress.
PreconditionsModules exist in the live roadmap.
PostconditionsStatus is updated and persisted.
Related FRsFR-407–409
Warn on Dependency Gaps
ActorsAI Engine
DescriptionWhen a user enters a module whose logical prerequisites are incomplete, the AI displays an advisory warning identifying the missing dependencies. The warning does not block the user.
PreconditionsUser attempts to enter a module with incomplete dependencies.
PostconditionsWarning is displayed. User can dismiss and proceed.
Related FRsFR-410
Suggest Module Additions
ActorsAI Engine
DescriptionAs new intake dimensions are captured, the AI evaluates whether the current roadmap has gaps and suggests additional modules if needed.
PreconditionsNew dimension information reveals a gap in the roadmap.
PostconditionsSuggestion is presented to the user for acceptance or dismissal.
Related FRsFR-412

3.5 Co-Pilot Module Work

Enter Module (Co-Pilot Mode)
ActorsUser, Database
DescriptionThe user clicks a module in the roadmap to enter co-pilot mode. The system displays the module header (name, knowledge area, description, inputs, outputs) and loads any existing conversation history.
PreconditionsThe module exists in the live roadmap.
PostconditionsModule view is rendered. Status updates to In Progress if it was Not Started.
Related FRsFR-505, FR-601, FR-408
Guide Module Conversation
ActorsAI Engine, User
DescriptionThe AI initiates and maintains a structured conversation tailored to the module type. It asks targeted questions, uses project context to avoid redundancy, and builds toward the module's expected outputs.
PreconditionsUser has entered co-pilot mode for a module.
PostconditionsUser has provided enough input for artifact generation.
Related FRsFR-602, FR-603
Generate Draft Artifact Content
ActorsAI Engine
DescriptionBased on the user's responses, the AI generates a structured draft of the module's artifact. The draft is presented for review, not saved automatically.
PreconditionsSufficient user input has been captured in the conversation.
PostconditionsDraft artifact content is displayed for user review.
Related FRsFR-604
Edit Artifact Content
ActorsUser
DescriptionThe user reviews and modifies any AI-generated content before saving. Edits can be made inline or through continued conversation with the AI.
PreconditionsDraft artifact content has been generated.
PostconditionsContent reflects the user's revisions.
Related FRsFR-605, FR-705
Flag Gaps & Suggest Improvements
ActorsAI Engine
DescriptionThe AI reviews the user's responses and draft content for weaknesses: missing stakeholders, vague requirements, unaddressed risks. It suggests specific improvements based on BABOK best practices.
PreconditionsDraft content exists or the conversation is in progress.
PostconditionsGap findings and suggestions are presented to the user.
Related FRsFR-606, FR-607
Save Artifact
ActorsUser, Database
DescriptionThe user saves the finalized artifact content. The system creates a versioned record linked to the project and module. Previous versions are preserved.
PreconditionsArtifact content has been reviewed and approved by the user.
PostconditionsArtifact record created in the database with version number.
Related FRsFR-701, FR-703, FR-706
Provide Completion Summary
ActorsAI Engine
DescriptionWhen the user indicates they are done with a module, the AI provides a summary of what was produced and flags any remaining gaps or areas that could be strengthened.
PreconditionsUser signals module completion.
PostconditionsSummary displayed. User can mark the module Complete or continue working.
Related FRsFR-610

3.6 Gap Analysis & Traceability

Run Gap Analysis
ActorsUser, AI Engine, Database
DescriptionThe user triggers an on-demand gap analysis. The AI reviews all completed artifacts for weaknesses, missing sections, vague language, and cross-artifact inconsistencies. Findings are presented as a prioritized, actionable list.
PreconditionsAt least one artifact has been completed.
PostconditionsGap analysis report is generated and displayed.
Related FRsFR-801, FR-802, FR-806
Generate Traceability Matrix
ActorsUser, AI Engine, Database
DescriptionThe system generates a traceability matrix linking business objectives to requirements, requirements to user stories, and user stories to acceptance criteria. The matrix auto-updates as artifacts change.
PreconditionsRequirements and related artifacts exist.
PostconditionsTraceability matrix is generated and displayed.
Related FRsFR-803, FR-804
Highlight Orphan Items
ActorsAI Engine
DescriptionThe AI identifies items in the traceability matrix that lack linkage: requirements without user stories, acceptance criteria without parent requirements, and similar orphans.
PreconditionsTraceability matrix exists.
PostconditionsOrphan items are flagged in the matrix view.
Related FRsFR-805

3.7 Export & Packaging

Export Individual Artifact
ActorsUser
DescriptionThe user exports a single artifact in a professional format (Markdown, PDF, or Word) with consistent branding including project name, engagement type, date, and artifact type.
PreconditionsThe artifact exists and has at least one saved version.
PostconditionsA formatted document is downloaded to the user's device.
Related FRsFR-901, FR-902, FR-904
Export Full Project Package
ActorsUser, Database
DescriptionThe user exports a complete deliverable package containing all selected artifacts, the traceability matrix, and a project summary cover page.
PreconditionsAt least one artifact exists. User selects which artifacts to include.
PostconditionsA packaged deliverable is downloaded.
Related FRsFR-903, FR-905

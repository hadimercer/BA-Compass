1. Overview

This document presents two UML diagrams for BA Compass: a Class Diagram documenting the core data model and entity relationships, and a Sequence Diagram documenting the intake classification interaction — the most complex and critical flow in the system.

Both diagrams are derived from the Product Definition Document v1.0 and the Functional Requirements Document v1.0. They serve as the structural and behavioral blueprints for the development phase.

2. Class Diagram — Core Data Model

The class diagram shows all persistent entities in BA Compass, their attributes, and the relationships between them. This directly maps to the PostgreSQL schema implemented in Neon DB.

Primary keys are marked with PK. Foreign keys are marked with FK. Data types shown are PostgreSQL types. Static reference tables (Module, RoadmapTemplate) are shaded blue.

2.1 Entity Descriptions

EntityTypeDescriptionKey Relationships
UserMutableAuthenticated user account. Custom auth with bcrypt password hashing and session token management.Owns 0..* Projects
ProjectMutableThe central object. All intake data, classification, roadmap, and artifacts scope to one Project.Belongs to 1 User. Has 0..* Dimensions, RoadmapItems, Artifacts, ConversationHistory
ProjectDimensionMutableOne row per intake dimension per project. Captured across three tiers with timestamp tracking.Belongs to 1 Project
ModuleStaticThe 52-module BABOK-aligned library. Read-only reference data seeded at deployment.Referenced by RoadmapItems and Artifacts
RoadmapTemplateStatic24 pre-built templates (8 types × 3 tiers). Each contains an ordered module_sequence as JSONB.References Modules via JSONB array
ProjectRoadmapItemMutableOne row per module in a project's live roadmap. Tracks sequence and status.Belongs to 1 Project. References 1 Module
ArtifactMutableStructured output from a module. Versioned — each save creates a new row. Content stored as JSONB.Belongs to 1 Project and 1 Module
ConversationHistoryMutablePer-module co-pilot conversation. Each row is one message (user or assistant role).Belongs to 1 Project and 1 Module

2.2 Key Design Decisions

DecisionRationale
UUIDs for all primary keysPrevents ID collision if data is ever merged or exported. Standard for modern web applications.
module_sequence as JSONB in RoadmapTemplateOrdered arrays of module IDs. Simpler than a join table for static reference data. Instantiated into ProjectRoadmapItem rows at runtime.
Artifact content as JSONBSupports varied artifact structures (tables, matrices, free-text) without schema changes per type.
Versioning via new rows, not updatesFull audit trail. Each artifact save creates a new version row. Latest version = max(version) per artifact_type per module.
user_id FK on all data tablesRow-level security foundation. Every query filters by authenticated user_id for data isolation.
ConversationHistory as separate tableConversation state must persist independently of artifacts so users can resume module work. Kept separate from artifacts for clean separation.

3. Sequence Diagram — Intake Classification

This sequence diagram shows the detailed message exchange between all four participants during the intake classification flow — the most complex interaction in BA Compass. It documents the exact order of calls, responses, decision points, and data persistence.

3.2 Interaction Fragments

FragmentTypeConditionDescription
loopLoopInsufficient signal for Must-Know dimensionsSteps 5–8 repeat until the AI has adequate confidence on Engagement Type, Scale Tier, Trigger/Origin, and Solution Clarity. Each iteration asks at most 2 questions.
altAlternativeUser accepts OR user overridesIf the user accepts the AI recommendation, classification proceeds as-is. If the user overrides, the system uses the user-selected type and tier instead. Both paths lead to roadmap generation.
